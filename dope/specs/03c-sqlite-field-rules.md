# SQLite: соглашения полей

## Идентификаторы

- `id` — внутренний SQLite primary key. Его можно показывать в debug, но не делать частью публичной схемы турнира.
- `tournament_id` — граница данных конкретного чемпионата. Запросы почти всегда фильтруются по нему.
- `scheme_id` — ссылка на JSON-схему, из которой создан турнир.

## Стабильные коды

- `slug` — стабильный человекочитаемый ключ для URL и импорта: `studchr-ek-2026`.
- `code` — короткий ключ внутри схемы: этап `r16`, бой `A`, пересев `after_r8`.
- `title` — отображаемый текст. Его можно менять без изменения `code`.

## Порядок и нумерация

- `*_index` — 0-based индекс для массивов и UI: `slot_index`, `theme_index`, `answer_index`.
- `roster_order` — 0-based порядок игрока в составе.
- `position` — 1-based порядок этапа или боя в сетке.
- `number` — публичный номер площадки: `1`, `2`, `3`.
- `place` и `rank` — спортивные места, 1-based; `place` может быть дробным для дележа.

## Ревизии и время

- `tournaments.revision` растет после любого изменения, видимого зрителю.
- `matches.revision` растет после изменения конкретного боя.
- `created_at`, `updated_at`, `applied_at` — RFC3339, лучше хранить в UTC.

## Статусы

Минимальный набор: `pending`, `active`, `finished`. Для `stage_type=reseed` статус `finished` означает, что `reseed_entries` рассчитаны для текущих результатов источников.

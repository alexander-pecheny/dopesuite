# SQLite: соглашения полей

## Идентификаторы

- `id` — внутренний SQLite primary key. Его можно показывать в debug, но не делать частью публичной схемы турнира.
- `tournament_id` — граница данных конкретного чемпионата. Запросы почти всегда фильтруются по нему.
- `game_id` — граница данных конкретной игры внутри турнира. Все таблицы бракета (стадии, бои, слоты, темы, ответы, результаты, пересевы) ведут к `game_id`.
- `scheme_id` — ссылка на JSON-схему, из которой создан бракет игры; `games.scheme_json` хранит снапшот.
- `user_id` — ссылка на учетную запись в `users`.

## Стабильные коды

- `slug` — стабильный человекочитаемый ключ для URL и импорта: `studchr-ek-2026`.
- `code` — короткий ключ внутри схемы: этап `r16`, бой `A`, пересев `after_r8`, игра `default`. Уникален в рамках своего родителя (`game_id` для стадий и боев, `tournament_id` для игр).
- `title` — отображаемый текст. Его можно менять без изменения `code`.
- `username` — публичный никнейм пользователя; уникален и не меняется после установки.

## Порядок и нумерация

- `*_index` — 0-based индекс для массивов и UI: `slot_index`, `theme_index`, `answer_index`.
- `roster_order` — 0-based порядок игрока в составе.
- `position` — 1-based порядок этапа, боя или игры в списке.
- `number` — публичный номер площадки или номер участника в корзине: `1`, `2`, `3`. Здесь же — `number` из `game_assignments`.
- `basket` — 1-based номер корзины жеребьевки.
- `place` и `rank` — спортивные места, 1-based; `place` может быть дробным для дележа.

## Внешние идентификаторы

- `tournaments.rating_id` — целочисленный ID на `rating.chgk.info/tournament/<rating_id>`. Опциональный. Не валидируется по сети, но клиенты могут отрисовывать ссылку.

## Аутентификация и сессии

- `invites.code` — короткий человекочитаемый код приглашения. Регистронезависимый при сравнении, хранится как ввели.
- `telegram_login_codes.code` — короткий код, который пользователь отправляет боту. Жизнь — 5 минут (см. [08-auth.md](08-auth.md)).
- `sessions.token_hash` — SHA-256 от cookie-токена; сам токен в БД не хранится.

## Ревизии и время

- `tournaments.revision` растет после любого изменения, видимого зрителю.
- `games.revision` растет после изменений в конкретной игре.
- `matches.revision` растет после изменения конкретного боя.
- `created_at`, `updated_at`, `applied_at`, `expires_at`, `consumed_at`, `last_seen_at` — RFC3339, лучше хранить в UTC.

## Статусы

Минимальный набор: `pending`, `active`, `finished`. Для `stage_type=reseed` статус `finished` означает, что `reseed_entries` рассчитаны для текущих результатов источников. Для `games.status` те же три значения.

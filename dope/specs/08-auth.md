# Учет пользователей и логин через Telegram

Открытой регистрации нет. Войти на сайт можно только по приглашению. Telegram-бот выступает sidecar'ом и используется как второй канал для одноразовых кодов: при регистрации пользователь несет код с сайта в бот, при логине — наоборот.

## Сущности

- `users` — учетная запись. До первого логина `username` может быть NULL. После того как пользователь установил `username`, его нельзя изменить.
- `invites` — одноразовое приглашение. Создается существующим пользователем, имеет срок жизни (по умолчанию 7 дней) и одного потребителя.
- `telegram_login_codes` — одноразовый код. Срок жизни — 1 минута. Поле `kind` различает `register` (привязка нового аккаунта по инвайту) и `login` (создание сессии для уже привязанного аккаунта). Для `register` строку выпускает сайт и потребляет бот; для `login` — наоборот.
- `sessions` — серверная сессия, кука хранит случайный токен, БД — только SHA-256 от него. Срок жизни — 30 дней с момента последнего обращения (sliding).

## Системный пользователь

Бутстрап создает запись `users` с `is_system=1`, `username='system'`. Системный пользователь является `created_by` для дефолтного турнира; ни один логин-код или сессию для него выдать нельзя.

## Поток A: регистрация по инвайту

1. Пользователь получает короткий код приглашения вне приложения (через автора инвайта).
2. На странице `/register` пользователь вводит инвайт-код. Сервер проверяет: запись существует, не использована, не истекла.
3. Сервер создает строку в `telegram_login_codes` с `kind='register'`, `invite_id`, новой `code` и `expires_at = now + 1min`. Возвращает код пользователю.
4. UI показывает код и просит отправить его боту @<bot_name>.
5. Пользователь пишет код в чат с ботом (просто текстом).
6. Бот находит строку по `code` (kind='register', не consumed, не expired), фиксирует `telegram_user_id` и `telegram_username` отправителя, ставит `consumed_at`. Отвечает «Готово».
7. UI ждет (long-poll или SSE) подтверждения. Сервер видит `consumed_at`, помечает инвайт как использованный, создает `users(telegram_user_id, telegram_username, username=NULL, is_system=0)` и сессию. Кука выдается на ответе.
8. UI обнаруживает сессию и переходит в шаг «выбери username»: пользователь вводит уникальный никнейм. Сервер делает один атомарный апдейт `update users set username=? where id=? and username is null`. Если конкуренция — повтор. После этого `username` иммутабелен.

## Поток B: логин уже зарегистрированного пользователя

1. Пользователь открывает `/login` на сайте — там одно поле «Код».
2. Пользователь пишет боту команду `/login`. Бот находит сендера по `telegram_user_id` в `users` (с `is_system=0`). Если не нашел — отвечает «Сначала зарегистрируйся по инвайту».
3. Бот создает строку в `telegram_login_codes` с `kind='login'`, заполненными `user_id`, `telegram_user_id`, `telegram_username`, новым `code` и `expires_at = now + 1min`. Отправляет код пользователю.
4. Пользователь вводит код на сайте. Сервер находит строку по `code` (kind='login', не consumed, не expired), помечает `consumed_at` и создает сессию для соответствующего `user_id`. Кука выдается на ответе.

## Команда `/start`

Бот отвечает короткой инструкцией со ссылками на `https://dope.pecheny.me/register` (для новых) и `https://dope.pecheny.me/login` (для уже зарегистрированных). Аргументы игнорируются.

## Сессия

- Кука: `session=<токен>`, `HttpOnly`, `Secure` в проде, `SameSite=Lax`. Сам токен — 32 случайных байта в hex.
- В БД: `sessions.token_hash = sha256(token)`. На каждом запросе `last_seen_at = now`.
- Логаут: удалить строку.

## Telegram-бот: интерфейс

В коде бот сидит за интерфейсом:

```go
type TelegramBot interface {
    Send(chatID int64, text string) error
    // Updates возвращает входящие сообщения; в проде — long-poll, в тестах — канал, который наполняет тест.
    Updates() <-chan TelegramUpdate
}
```

Реализация в `dope/cmd/telegram-bot` делает long-poll к Telegram Bot API и работает с тем же файлом БД, что и сервер. Без `TELEGRAM_BOT_TOKEN` бот стартует в idle-режиме (no-op), чтобы локальный `just dev` не требовал токена.

## Безопасность и инварианты

- Инвайт-код одноразовый и истекает (по умолчанию 7 дней).
- Логин/регистр-коды одноразовые, истекают за 1 минуту, привязаны к одному `kind`.
- `username` уникален и иммутабелен; нет UI или хендлера, который его меняет.
- Сессионные токены никогда не логируются и не возвращаются повторно после первой выдачи.
- `users.telegram_user_id` уникален.
- Системный пользователь `is_system=1` не получает сессий.
- Турнирные API не доверяют host/viewer UI: публичные турниры доступны на чтение, приватные — только организаторам, все изменения требуют сессию организатора из `tournament_organizers`.
- Небезопасные API-запросы с сессионной кукой дополнительно проверяют `Origin`, если браузер его прислал.

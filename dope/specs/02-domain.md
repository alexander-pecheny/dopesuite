# Доменная модель

## Учет пользователей

- `User` — учетная запись: `username` (уникальный, неизменяемый), привязка к Telegram, флаг `is_system`. Регистрация только по приглашению.
- `Invite` — одноразовый код приглашения, выпускает существующий пользователь.
- `TelegramLoginCode` — короткоживущий одноразовый код, который пользователь отправляет sidecar-боту, чтобы привязать Telegram-аккаунт (на регистрации) или создать сессию (на входе).
- `Session` — серверная сессия. Кука несет случайный токен, в БД хранится только хэш.

См. [08-auth.md](08-auth.md) для протокола регистрации/входа.

## Турнир и игры

- `Tournament` — соревнование верхнего уровня: `id`, опциональный `rating_id` (ссылка на `rating.chgk.info/tournament/<rating_id>`), `title`, многострочное markdown-описание, `created_by`, список организаторов, опциональные `start_date`/`end_date` (`YYYY-MM-DD`), флаг `is_public` (по умолчанию 0). Только публичные турниры показываются на `/`; приватные доступны организаторам через `/host/tournament/{id}`.
- `TournamentOrganizer` — пользователь, который может быть ведущим и редактировать состав турнира.
- `Game` — одна игра внутри турнира. У игры свой `game_type` (`ek`, `brain`, `si`, `od`, ...), свой бракет (импортированная JSON-схема), свой список команд/игроков, своя соответствие команда↔игрок и команда/игрок↔корзина+номер. На текущей итерации реализован только `ek`; остальные форматы добавятся позже со своими таблицами состояния матча.
- `Stage`, `Match`, `MatchSlot` — теперь принадлежат конкретной `Game`, а не напрямую турниру. Тип этапа — `matches` или `reseed`, как и раньше. У слота место в протоколе фиксируется `slot_index`, а назначение участника происходит через бракет (`seed`/`from_match`/`reseed`/`placeholder`).
- `Assignment` — соответствие `(game_id, basket, number) → team_id | player_id`. Это и есть мост между символами в JSON-сетке и конкретными участниками. Бракет JSON больше не содержит конкретных команд.

См. [09-tournaments-and-games.md](09-tournaments-and-games.md) для полей и наследования.

## Команды и игроки

- `Team` — команда турнира: `id`, `name`, `city`. Принадлежит турниру.
- `Player` — игрок турнира: `id`, `first_name`, `last_name`. Принадлежит турниру.
- `TeamPlayer` — состав команды на уровне турнира, до 9 игроков.

Игра по умолчанию использует команды/игроков и составы турнира. Если хост хочет переопределить (например, игрок меняет команду между играми), он переключает у игры флаг наследования (`team_list_source` или `roster_source` со значения `'tournament'` на `'game'`). При переключении система копирует текущее состояние турнира в game-уровневые таблицы; дальше турнирные правки не доходят до этой игры, пока флаг снова не переключат.

## Площадки

- `Venue` — площадка турнира: номер и название. Площадки общие для всего турнира; конкретный бой ссылается на `venue_id`, и его можно поменять без последствий для счета.

## Источники участника боя

В JSON-сетке слот может ссылаться только на одно из четырех:

- `seed` — корзина+номер. Разрешается через `game_assignments`.
- `fromMatch` — место в предыдущем бою. Разрешается через `match_results` источника.
- `reseed` — конкретный ранг этапа-пересева. Разрешается через `reseed_entries`.
- `placeholder` — временная подпись до разрешения источника.

Источник `team` (явная команда внутри слота) больше не поддерживается: бракет — это спортивная схема, а не назначение команд.

## Жизненный цикл

1. Пользователь логинится (см. 08-auth).
2. Пользователь создает турнир и добавляет в него игры (импортируя JSON-сетки).
3. Хост заполняет `game_assignments` — кто стоит на какой корзине+номере.
4. Резолвер вычисляет `match_slots.team_id` (или `player_id` для индивидуальных игр) из источника слота: `seed` → `game_assignments`; `fromMatch` → результаты предыдущего боя; `reseed` → `reseed_entries`.
5. Ведущий заполняет бой: игроки, ответы, места, перестрелка.
6. Когда игровой этап завершен, следующий `reseed`-этап считает ранги и заполняет `reseed_entries`. Зависящие слоты разрешаются автоматически.
7. Площадку можно изменить в любой момент.

## Статистика

Для EK-игр по-прежнему: для каждой команды в бою — место, сумма, `Σ+`, перестрелка, количества взятых/невзятых вопросов по номиналам. Для пересевов агрегируется по выбранным этапам. Другие `game_type` будут хранить состояние и считать статистику по своим правилам.

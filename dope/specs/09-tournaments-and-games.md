# Турниры и игры

Турнир — это контейнер для одной или нескольких игр. Каждая игра живет по своему бракету и имеет своих участников; команды и игроки по умолчанию приходят из турнира.

## Поля турнира

- `id` — внутренний ID.
- `slug` — стабильный человекочитаемый ключ для URL.
- `title` — отображаемое название.
- `description` — markdown, многострочный. Рендерится goldmark в безопасном режиме (raw HTML экранируется).
- `rating_id` — опциональное целое, ссылка на `https://rating.chgk.info/tournament/<rating_id>`. UI рендерит ссылку, если поле заполнено.
- `start_date`, `end_date` — `YYYY-MM-DD`, опциональны.
- `is_public` — флаг показа на публичной странице `/`. По умолчанию 0.
- `created_by` — `users.id` создателя; автоматически добавляется в организаторы.
- `tournament_organizers` — список пользователей, которые могут редактировать турнир, его игры и проводить бои.

## Поля игры

- `id`, `tournament_id`, `code`, `title`, `position`.
- `game_type` — `ek` (реализован), `brain`, `si`, `od`, ... Определяет, какие таблицы состояния матча используются.
- `scheme_id` + `scheme_json` — JSON-бракет, импортированный для этой игры. `scheme_json` — снапшот, чтобы изменения исходной схемы не ломали уже идущую игру.
- `status` — `pending` / `active` / `finished`.
- `team_list_source` — `'tournament'` или `'game'`.
- `roster_source` — `'tournament'` или `'game'`.
- `revision` — растет при любом изменении игры.

### Состав команды/игроков

- В командной игре список команд и составов берется по правилам:
  - `team_list_source='tournament'`: список команд = команды турнира (`teams.tournament_id = tournament_id`).
  - `team_list_source='game'`: список команд = `game_teams(game_id)`.
  - `roster_source='tournament'`: состав каждой команды = `team_players(team_id)`.
  - `roster_source='game'`: состав = `game_team_players(game_id, team_id)`.
- В индивидуальной игре список участников = `game_players(game_id)`. Турнир-уровневого списка индивидуальных участников пока нет; когда появится — эта таблица станет необязательной (как `game_teams`).

### Переключение наследования (override)

Когда хост переключает `team_list_source` или `roster_source` со `tournament` на `game`, сервер делает «снапшот»:

1. Считывает текущее наследуемое состояние (команды турнира или составы команд турнира).
2. Записывает в `game_teams` / `game_team_players` соответствующие строки.
3. Меняет флаг.

Дальнейшие правки на турнирном уровне не влияют на игру, пока флаг не вернут на `tournament`. Возврат флага удаляет override-строки и возвращает живое наследование.

### Соответствие команды/игрока корзине+номеру

- Бракет JSON ссылается на участника символично: `{"seed": {"basket": K, "number": N}}`.
- Конкретное соответствие хранится в `game_assignments(game_id, basket, number, team_id?, player_id?)`.
- В командной игре заполняется `team_id`, в индивидуальной — `player_id`. CHECK-ограничение требует, чтобы ровно одно из них было не NULL.
- Для одного `(game_id, basket, number)` может быть только одна запись.
- Хост редактирует `game_assignments` через UI игры (страница «жеребьевка»). Резолвер слотов читает только эту таблицу для источника `seed`.

## Поток создания турнира и игр

1. Авторизованный пользователь жмет «новый турнир», заполняет `title`, `description`, опциональный `rating_id`.
2. После создания пользователь становится `created_by` и единственным организатором; может позвать других.
3. Организатор добавляет игру: выбирает `game_type`, импортирует JSON-бракет (или клонирует существующую сетку).
4. Игра создается с `team_list_source='tournament'`, `roster_source='tournament'`, пустым `game_assignments`. Резолвер слотов на этом этапе не находит команд для `seed` — слоты остаются «не разрешенными» в UI.
5. Организатор вводит соответствия в `game_assignments`. Слоты разрешаются автоматически.
6. По мере проведения боев `from_match`/`reseed` источники разрешаются по результатам.

## Что остается прежним

- Площадки (`venues`) живут на уровне турнира и общие для всех игр турнира.
- EK-специфичные таблицы (`themes`, `answers`, `match_results`, `reseed_entries`) не меняют форму. Только их родитель `matches` теперь знает, к какой игре относится.
- API имеет дефолтную игру дефолтного турнира до тех пор, пока хендлеры не получат явные `tournament_id`/`game_code` (см. [07-implementation.md](07-implementation.md)).

# SQLite

База: `tournament.db`. Включаем `PRAGMA foreign_keys = ON`, желательно WAL. Миграции версионируются.

## Разделы

- [03a-sqlite-core.md](03a-sqlite-core.md) — схема, турнир, команды, игроки, площадки, этапы, бои.
- [03b-sqlite-match.md](03b-sqlite-match.md) — слоты, темы, ответы, результаты, пересевы, SSE-события.
- [03c-sqlite-field-rules.md](03c-sqlite-field-rules.md) — общие соглашения по полям.
- [03d-sqlite-match-fields.md](03d-sqlite-match-fields.md) — подробности боевых полей, включая `slot_index`.

## Принципы

- JSON-схема хранится целиком в `schemes.schema_json`; таблицы хранят конкретное состояние турнира.
- Все таблицы, кроме версий миграций, относятся к конкретному `tournament_id` напрямую или через бой.
- `stages` хранит и игровые этапы, и этапы-пересевы; тип задает `stage_type`.
- `matches.venue_id` — текущая площадка боя; исходный дефолт берется из JSON-схемы при импорте.
- `match_slots.team_id` — уже разрешенная команда. Пока источник неизвестен, слот остается с `team_id = null`.
- Результаты можно пересчитать из ответов, но `match_results` хранит кеш для сетки, пересевов и быстрых API.
- `events` хранит ревизии для SSE и диагностики, но не является единственным источником правды.

## Инварианты приложения

- Состав команды: не больше 9 строк в `team_players`.
- В бою количество слотов равно `matches.participant_count`.
- В обычном бою ровно `regularThemeCount` тем на команду; перестрелка может добавляться динамически.
- `mark` принимает только `''`, `right`, `wrong`.
- Поля с суффиксом `_index` — внутренние 0-based индексы; `place`, `rank`, `number`, `position` — 1-based или человекочитаемые.
